*** [2026-04-17] – Run =git ls-tree= from =git_root= (not =skill_dir=) so pathspec is unambiguously repo-root-relative
**** Reason
=git ls-tree= pathspecs are interpreted against the tree object, not cwd; running it from
=skill_dir= with a pathspec of =my-skill= would be ambiguous.  Running from =git_root= with the
pathspec as =relative(git_root, skill_dir)= removes the ambiguity entirely.  Output paths
(repo-root-relative) are then stripped of the skill prefix to yield skill-dir-relative
=source_name=.  =git cat-file blob <hash>= is hash-based and needs no particular cwd, but also
runs in =git_root= for uniformity across the function.
**** Alternatives considered
- Run =ls-tree= from =skill_dir= without a pathspec, then filter by prefix client-side: wasteful
  in large monorepos (reads entire tree), and the filtering logic duplicates what git already
  does via pathspec.

*** [2026-04-17] – Empty subtree at valid ref returns =[]=; =validate_manifest= surfaces the error
**** Reason
When a ref resolves but =skill_dir= has no files at that ref (e.g. the directory didn't exist
yet), =git ls-tree= exits 0 with empty output.  The function returns =[]= silently; the
existing =validate_manifest= then reports "No entry targets SKILL.md" with an actionable
message.  Keeps =build_manifest_from_git_ref= honest about what git said and avoids duplicating
validation logic.
**** Alternatives considered
- Throw a dedicated "no files at ref" error: duplicates =validate_manifest='s job; two errors
  for the same user-facing problem.

*** [2026-04-17] – =tsconfig.json= =lib= must match a lib file shipped by the installed TypeScript
**** Reason
TypeScript 5.9.3 ships lib files up to =ES2024=.  Using =lib: ["ES2025"]= was silently accepted
by =tsgo= (the Go preview checker used by =npm run typecheck=) but rejected by the classic =tsc=
that powers the editor's language server — which fell back to a default that excluded ES2023
array methods, breaking =toSorted= and cascading into implicit-=any= errors throughout the
file.  Switched to =lib: ["ES2024"]= so both checkers agree.  =RegExp.escape= (Stage 3 proposal,
not yet in any TS lib) is suppressed with a single =@ts-expect-error=.
**** Alternatives considered
- Replace =toSorted= with =[...arr].sort=: works, but loses the cleaner idiom and doesn't fix
  the root cause (invalid =lib= value still accepted silently by =tsgo=).

*** [2026-04-10] – Use stage > 0 entries to detect index conflicts; abort =--staged= with generic "conflicts" message
**** Reason
During merge, rebase, cherry-pick, revert, and =git am=, git writes stage-1/2/3 entries to the
index for conflicted files instead of a single stage-0 entry.  Checking for any stage > 0 entry
(from the =git ls-files --stage= output already being read) detects all conflict-producing operations
without enumerating =.git/= sentinel files (=MERGE_HEAD=, =rebase-merge/=, etc.).  The check is
naturally scoped to =skill_dir= via the existing pathspec, avoiding false positives in monorepos
where a conflict exists in an unrelated subtree.  Error message says "conflicts" (not "merge conflicts")
since the cause may be any git operation.
**** Alternatives considered
- Detect in-progress state via =.git/MERGE_HEAD=, =rebase-merge/= etc.: requires maintaining a
  complete enumeration of sentinel paths; is repo-wide (false positives in monorepos); misses future
  git operations that produce conflicts.

*** [2026-04-02] – =--clean= cleanliness check scoped to =skill_dir= subtree, not the whole repo

*** [2026-03-30] – Atomicity: sentinel file approach; no write-to-temp-then-rename
**** Reason
A crash mid-write-loop leaves some target files at new content and some at old,
with the sidecar still recording old hashes — spurious hash-mismatch error on
next deploy.  Fix: write a ~.skill-shed-deploy-in-progress~ sentinel at the
start of the write loop, delete it after ~write_sidecar~.  On next deploy,
~deploy()~ checks for the sentinel before the overwrite check; if present, exits
with code 1 telling the user to re-run with ~--force~.
Disk-full failures are considered negligible (skills are small).
Write-to-temp-then-rename (atomic sidecar write) was considered but rejected:
it only fixes the case where all file writes succeeded but the sidecar write
itself was interrupted — a much narrower fix that still leaves the mid-loop
crash unaddressed.
**** Alternatives considered
- Ignore entirely, rely on ~--force~ for recovery: confusing error message gives
  no hint that interrupted deploy is the cause.
- Write sidecar first, then files: still produces mismatches (new hashes, old
  file content after crash).
- Write to temp then rename (atomic sidecar): only fixes torn-sidecar-write,
  not mid-loop crash.
- Write all files to a temp directory, then rename the whole directory into
  place: truly atomic, but ~rename~ across directories is not atomic on all
  filesystems, and replacing a live target directory atomically requires
  platform-specific tricks (~renameat2~ on Linux, not available in Node's
  ~fs.rename~).  Significant complexity for a tool that deploys small text files.
- Per-file write-to-temp-then-rename: atomic per file, but does not help if
  the process crashes between files; target still ends up in a mixed state.
  More code, same fundamental problem.
- Revert partially successful deploy on next run: save the previous content of
  each target file before overwriting, then restore on interrupted-deploy
  detection.  Truly recovers the pre-deploy state, but requires storing backups
  of all target files (doubling storage), significantly more code, and adds a
  new failure mode (crash during the revert itself).  Overkill for small text
  files where re-deploying with ~--force~ is trivially fast.

*** [2026-03-26] – Manifest is an array of ManifestEntry objects, not a Map
**** Reason
Iteration is the dominant operation; lookup by name is rare and only needed
for ~5–20 entries (skill files are small).  The conflict check
(~SKILL.md~ vs ~SKILL.source.md~ both resolving to the same target) requires
an explicit scan regardless of container — a ~Map~ keyed by target name would
silently overwrite and still needs a pre-check.  Since the ~Map~ invariant
doesn't help here, an array is the more honest representation: an ordered
collection of records with no artificial primary key.  A ~validate_manifest~
function checks for duplicate ~target_name~s explicitly, replacing the
implicit (but non-functional) ~Map~ guarantee.
**** Alternatives considered
- ~Map<string, ManifestEntry>~ keyed by ~target_name~: doesn't enforce
  uniqueness for the source-conflict case; O(1) lookup irrelevant at n ≤ 20.

*** [2026-03-26] – ManifestEntry shape: source_name, target_name, source_content, target_content, line_map
**** Reason
All transform types (comment stripping, Org→md, AsciiDoc→md) produce a
source and a target.  Storing both in one entry gives linting access to
the source and deploy access to the target without reconstruction.
~source_content~ is ~string | Buffer~ (string for ~.md~/~.source.md~,
Buffer for ~.org~ etc.); ~target_content~ is always ~string~ (always ~.md~
after any transform).  ~line_map~ is optional: only meaningful when
~source_content~ is a string (comment-stripping path); undefined for
pandoc-converted sources.  Pass-through files have
~source_name === target_name~ and ~source_content === target_content~.
**** Alternatives considered
- Store only ~target_content~: loses source for linting and line mapping.
- Store ~source_content~ only for transformed files: inconsistent shape,
  complicates consumers.

*** [2026-03-26] – Manifest stores .md files as strings, all others as Buffers
**** Reason
~Buffer → string~ via UTF-8 is not lossless for binary files (invalid byte
sequences are replaced with U+FFFD), so storing binary files as strings is
wrong.  A null-byte heuristic for binary detection is unreliable and puts
content-type logic in the wrong layer.  Extension-based detection is
deterministic and correct: ~*.md~ files (including ~*.source.md~) are always
text; everything else is kept as ~Buffer~.  No heuristic needed; the manifest
already has the filename.
**** Alternatives considered
- ~Buffer~ for all files, decode at use site: scatters ~.toString('utf8')~ calls
  across all consumers; punishes the common (text) case.
- Null-byte heuristic: unreliable; misclassifies valid UTF-8 with embedded nulls.
- ~string~ for all files via ~latin1~: technically lossless but unnatural and
  requires ~latin1~ encode/decode at every binary use site.

*** [2026-03-25] – Replace mtime guard with content-hash sidecar `.skill-shed-manifest.json`
**** Reason
Mtime is unreliable in a git workflow: index objects have no mtime; `git
checkout`/`stash pop` reset working-tree mtimes to now; partially staged
files have a working-tree mtime that differs from the staged content.
Content hashing is immune to all of these.  The sidecar records SHA-256
hashes of every file as deployed; next deploy compares current target
hashes against stored hashes to detect direct edits.
**** Alternatives considered
- Keep mtime for single-file and add hashing only for multi-file: two code
  paths, inconsistent semantics.

*** [2026-03-25] – Sidecar `.skill-shed-manifest.json` lives in the target directory
**** Reason
The sidecar guards the target, so it belongs there.  The overwrite check
can be performed without knowing the source path or loading ~.env~.  If
the source directory is moved or deleted the target remains protected.
**** Alternatives considered
- Source dir: requires resolving target path before the check can run;
  orphaned if source dir is moved.

*** [2026-03-25] – Per-file overwrite safety check against sidecar
**** Reason
For each file to be deployed, three cases arise:
1. Target file absent: write freely (genuinely new).
2. Target file present, sidecar has entry: hash check — abort if hashes
   differ (direct edit detected), otherwise overwrite.
3. Target file present, sidecar has no entry (sidecar absent or file not
   listed): abort unless ~--force~ (file may have been manually created).
The deploy aborts as a whole on any case-2 or case-3 hit; partial deploys
leave the target in an inconsistent state.  ~--force~ bypasses both the
missing-sidecar check and a hash-mismatch check (same flag, same
semantics: "I accept responsibility for any overwritten edits").
**** Alternatives considered
- Abort only when sidecar is entirely absent: misses the case where a new
  file is added to an existing multi-file skill after first deploy.
- Abort only the conflicting file, deploy the rest: partial deploy leaves
  target inconsistent; --force covers the intentional-overwrite use case.

*** [2026-03-25] – Stale-file policy: delete owned files, abort on unowned files
**** Reason
A file present in the target but absent from the current deploy set falls
into one of two cases, distinguishable via the sidecar:
- In sidecar: skill-shed deployed it; it was deleted in the new version.
  Safe to auto-delete.
- Not in sidecar: not deployed by skill-shed; origin unknown.  Abort
  unless ~--force~ (conservative; may warrant further thought).
**** Alternatives considered
- Warn instead of abort on unowned files: leaves target in mixed state,
  harder to reason about.
- Ignore unowned files: silently accumulates cruft.
