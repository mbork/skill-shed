** Decisions
*** [2026-03-26] – Manifest stores .md files as strings, all others as Buffers
**** Reason
~Buffer → string~ via UTF-8 is not lossless for binary files (invalid byte
sequences are replaced with U+FFFD), so storing binary files as strings is
wrong.  A null-byte heuristic for binary detection is unreliable and puts
content-type logic in the wrong layer.  Extension-based detection is
deterministic and correct: ~*.md~ files (including ~*.source.md~) are always
text; everything else is kept as ~Buffer~.  No heuristic needed; the manifest
already has the filename.
**** Alternatives considered
- ~Buffer~ for all files, decode at use site: scatters ~.toString('utf8')~ calls
  across all consumers; punishes the common (text) case.
- Null-byte heuristic: unreliable; misclassifies valid UTF-8 with embedded nulls.
- ~string~ for all files via ~latin1~: technically lossless but unnatural and
  requires ~latin1~ encode/decode at every binary use site.

*** [2026-03-26] – Manifest as abstraction; MVP uses build_manifest_from_dir
**** Reason
The manifest (a ~Map~ of filename → content/Buffer) is a stable abstraction.
The MVP implements it as ~build_manifest_from_dir~: ~readdir~ + dotfile filter
(~10 LoC).  Git support lands as =build_manifest_from_git= and replaces it.
No code is thrown away; the interface is unchanged.
Dotfile exclusion (already the design decision) covers ~.env~ as a side effect,
eliminating the safety concern raised against a ~readdir~-based approach.
=*~= backup files will be deployed by the MVP — ugly but harmless; addressed
when ~build_manifest_from_git~ lands.
**** Overrides
- [[*\[2026-03-24\] – File manifest via git ls-files --exclude-standard (run from skill dir); MANIFEST_CMD overrides in .env]]
  (deferred to a later step; git integration is ~build_manifest_from_git~)
**** Alternatives considered
- Skip MVP, implement git manifest directly: more complex first step, no benefit
  given the abstraction makes the swap trivial.

*** [2026-03-24] – File manifest via git ls-files --exclude-standard (run from skill dir); MANIFEST_CMD overrides in .env
NOTE: superseded by [[*\[2026-03-26\] – Manifest as abstraction; MVP uses build_manifest_from_dir]]
**** Reason
`git ls-files --exclude-standard` run from the skill directory naturally
scopes to the skill subtree (monorepo-safe) and respects `.gitignore`,
`.git/info/exclude`, and the global gitignore — so `.env` and other
gitignored files are excluded without hardcoding.  MANIFEST_CMD in `.env`
allows non-git workflows.  Note: `--exclude-standard` only excludes
untracked files; if `.env` is accidentally committed it will still appear
— the author must ensure it is gitignored and never committed.
**** Alternatives considered
- Explicit manifest file: redundant when git is available.
- `deploy/` subdirectory convention: requires restructuring skill directories.
- Hardcoded exclusion list: whack-a-mole, always incomplete.

*** [2026-03-24] – Dotfiles excluded by default; DEPLOY_DOTFILES whitelist in .env for exceptions
**** Reason
`git ls-files` alone would deploy `.gitignore` and other dotfiles that are
tracked but not intended for deployment.  Dotfile exclusion as a second
filter is simple and covers the common case.  DEPLOY_DOTFILES allows
opt-in for dotfiles that are legitimate skill assets (e.g. template
dotfiles).  The two filters are orthogonal: deploy if in `git ls-files`
AND (not a dotfile OR whitelisted).
**** Alternatives considered
- Hardcoded exclusion list: incomplete, requires maintenance.
- `deploy/` subdirectory: principled but requires restructuring.
- No dotfile rule: `.gitignore` and similar files would be deployed.

*** [2026-03-24] – Two deploy modes: production (default) and development
**** Reason
Production enforces a fully clean repo (abort on any deviation: staged,
unstaged, or untracked non-ignored files).  Development supports iterative
workflows by deploying the index when index ≠ HEAD, otherwise deploying
the working tree.  No confirmation message in the clean case.
**** Alternatives considered
- Single mode: too strict for iterative dev or too loose for production.

*** [2026-03-24] – Development mode deploy source: index if index ≠ HEAD, else working tree
**** Reason
One binary check, no per-file branching.  When anything is staged, the
index is the most intentional snapshot.  When nothing is staged, the
working tree reflects current edits.
**** Alternatives considered
- Always deploy working tree: ignores staged-only changes; deploys
  working-tree-only changes even when index differs from HEAD elsewhere.
- Per-file decision: complex and harder to reason about.

*** [2026-03-24] – Reporting logic for development mode
**** Reason
Deploy index (index ≠ HEAD): list all deployed files; separately list
files with unstaged changes not included in the deploy (modified, deleted
from working tree, untracked non-ignored).  Note: a partially staged file
may appear in both lists.
Deploy working tree (index == HEAD, working tree ≠ HEAD): list all
deployed files marking changed ones (includes untracked non-ignored);
separately list files in HEAD deleted from working tree (skipped).
Clean: no output.

*** [2026-03-25] – Replace mtime guard with content-hash sidecar `.skill-shed-manifest.json`
**** Reason
Mtime is unreliable in a git workflow: index objects have no mtime; `git
checkout`/`stash pop` reset working-tree mtimes to now; partially staged
files have a working-tree mtime that differs from the staged content.
Content hashing is immune to all of these.  The sidecar records SHA-256
hashes of every file as deployed; next deploy compares current target
hashes against stored hashes to detect direct edits.
**** Alternatives considered
- Keep mtime for single-file and add hashing only for multi-file: two code
  paths, inconsistent semantics.

*** [2026-03-25] – Sidecar `.skill-shed-manifest.json` lives in the target directory
**** Reason
The sidecar guards the target, so it belongs there.  The overwrite check
can be performed without knowing the source path or loading ~.env~.  If
the source directory is moved or deleted the target remains protected.
**** Alternatives considered
- Source dir: requires resolving target path before the check can run;
  orphaned if source dir is moved.

*** [2026-03-25] – Per-file overwrite safety check against sidecar
**** Reason
For each file to be deployed, three cases arise:
1. Target file absent: write freely (genuinely new).
2. Target file present, sidecar has entry: hash check — abort if hashes
   differ (direct edit detected), otherwise overwrite.
3. Target file present, sidecar has no entry (sidecar absent or file not
   listed): abort unless ~--force~ (file may have been manually created).
The deploy aborts as a whole on any case-2 or case-3 hit; partial deploys
leave the target in an inconsistent state.  ~--force~ bypasses both the
missing-sidecar check and a hash-mismatch check (same flag, same
semantics: "I accept responsibility for any overwritten edits").
**** Alternatives considered
- Abort only when sidecar is entirely absent: misses the case where a new
  file is added to an existing multi-file skill after first deploy.
- Abort only the conflicting file, deploy the rest: partial deploy leaves
  target inconsistent; --force covers the intentional-overwrite use case.

*** [2026-03-25] – Stale-file policy: delete owned files, abort on unowned files
**** Reason
A file present in the target but absent from the current deploy set falls
into one of two cases, distinguishable via the sidecar:
- In sidecar: skill-shed deployed it; it was deleted in the new version.
  Safe to auto-delete.
- Not in sidecar: not deployed by skill-shed; origin unknown.  Abort
  unless ~--force~ (conservative; may warrant further thought).
**** Alternatives considered
- Warn instead of abort on unowned files: leaves target in mixed state,
  harder to reason about.
- Ignore unowned files: silently accumulates cruft.

*** [2026-03-25] – `init` logic for existing directories
**** Reason
After several design iterations, the agreed `init` flow is:
1. If =skill_dir= exists, read its file list; else create it (file list = []).
2. If any two files share the same deploy target (=target_filename=): abort
   "Cannot initialize, conflicting files: {{list}}".
3. Set =does_skill_file_exist= = false.
4. If =SKILL.source.md= exists:
   - If =--no-comments=: abort "Cannot initialize =SKILL.md= when =SKILL.source.md= present".
   - Tell user "=SKILL.source.md= already exists"; set =does_skill_file_exist= = true.
5. Else if =SKILL.md= exists:
   - If =--comments=: abort "Cannot initialize =SKILL.source.md= when =SKILL.md= present".
   - Tell user "=SKILL.md= already exists"; set =does_skill_file_exist= = true.
6. If =.env= exists: abort "Cannot initialize, =.env= already present".
   Else: create =.env= and tell user "=.env= created".
7. If =not does_skill_file_exist=: create =SKILL.source.md= (default) or =SKILL.md= (=--no-comments=).

When no flag is passed and =SKILL.md= already exists, =init= accepts it without
aborting; =deploy= autodetects which source file is present.  This keeps =init=
non-destructive for re-initialization (e.g. adding =.env= to an existing skill dir).
**** Alternatives considered
- Abort when default (~--comments~) doesn't match existing file:
  rejected — unnecessarily obstructive for re-init workflows where the
  user just wants to add =.env=.
- Check =.env= before source-file conflicts: rejected — produces partial writes
  (=.env= created before a later abort fires).

*** [2026-03-25] – Built-in stripper implemented now instead of pandoc
**** Reason
Pandoc does not strip HTML comments by default; it also strips the YAML frontmatter by
default.  Adding a Lua filter would defeat the purpose of using pandoc for simplicity.
The built-in fenced-code-block-aware scanner (already planned in "Configurable comment
stripping") is therefore the immediate implementation path.
**** Alternatives considered
- Pandoc with Lua filter: rejected — pandoc behavior requires too many workarounds (frontmatter

*** [2026-03-24] – Strip in-memory, no intermediate SKILL.md; add --dry-run for inspection
**** Reason
An intermediate git-ignored file adds a third source of truth (source,
intermediate, target), is silently removed by `git clean`, and its mtime
guard is prone to false positives from editors/backup tools.  In-memory
stripping is simpler.  `--dry-run` covers the inspection use case without
the complexity.
**** Alternatives considered
- Intermediate SKILL.md (git-ignored): rejected — unreliable mtime guard,
  muddy mental model, removed by git clean.

*** [2026-03-24] – Source filename fixed as SKILL.source.md; format is HTML comments
**** Reason
Simplicity and consistency.  HTML comments are unambiguous and support
multiline blocks.  Auto-detected comment char (4 identical leading chars)
was rejected: `####` is a valid Markdown H4 heading and would be
misdetected; also line-comments only.
**** Alternatives considered
- Auto-detected comment char: rejected — collides with Markdown headings,
  line-comments only.

*** [2026-03-25] – `init` creates `SKILL.source.md` by default; `--no-comments` creates `SKILL.md`
**** Reason
Comment stripping is a core feature; making it the default improves discoverability and
establishes the right habit from day one.  `--no-comments` is available for users who don't
need it.  A single template file is used for both paths; the `--no-comments` path strips the
example comment block (explicitly delimited) before writing `SKILL.md`.  Pandoc is the
PoC-only stripper until the built-in implementation lands — the built-in stripper is a
hard prerequisite for any stability milestone.
**** Alternatives considered
- `init` creates `SKILL.md` by default (opt-in rename to `.source.md`): less discoverable,
  comment stripping is a second-class citizen.
- Two template files: rejected — maintenance burden, content drift risk.
- `--no-comments` flag name: clearer intent, but `--no-source` is consistent with the
  `.source.md` naming convention used throughout the tool.

*** [2026-03-24] – Strip HTML comments via pandoc, not in-script regex
**** Reason
Pandoc handles code blocks correctly (comments inside code blocks are
preserved as content, not stripped).  A naive regex would incorrectly
strip them.  A proper in-script Markdown parser is a significant
dependency — no better than pandoc.  Pandoc is already required for
alternative source formats, so no new dependency for mixed workflows.
**** Alternatives considered
- In-script regex: incorrect for comments inside code blocks.
- In-script Markdown parser: significant dependency, no advantage over pandoc.
